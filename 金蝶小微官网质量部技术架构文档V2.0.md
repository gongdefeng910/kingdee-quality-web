## 一、现有架构评审（V1.0 问题清单）

### 1.1 架构层面

| 问题编号 | 问题描述 | 风险等级 | 影响范围 |
|---------|---------|---------|---------|
| A-01 | CORS 完全开放（`CORS(app)` 无域名限制），任意域名可调用 API | 高 | 全部接口 |
| A-02 | 无 API 版本管理（`/api/login` 而非 `/api/v1/login`），后续迭代无法兼容 | 中 | 接口演进 |
| A-03 | SQLite 单文件数据库，不支持并发写入，Render 部署时文件系统为临时存储，重启后数据丢失 | 高 | 数据持久化 |
| A-04 | 前后端耦合度低但缺少统一配置管理，API 地址散落在 `main.js`、`admin.js`、`localStorage` 中 | 中 | 部署维护 |
| A-05 | 无数据库迁移策略（无 Migration 工具），表结构变更依赖手动 SQL | 中 | 版本迭代 |
| A-06 | 无日志采集与监控告警体系，线上问题无法追溯 | 高 | 运维排障 |
| A-07 | 无 API 网关或反向代理层，后端直接暴露 | 中 | 扩展性 |

### 1.2 安全层面

| 问题编号 | 问题描述 | 风险等级 | OWASP 分类 |
|---------|---------|---------|-----------|
| S-01 | 硬编码密钥 `kingdee-quality-secret-key-2026`（app.py 第19行） | **严重** | A02: 加密失败 |
| S-02 | 管理员凭据硬编码 `admin/admin123`（app.py 第131行），无密码哈希存储 | **严重** | A07: 身份识别和认证失败 |
| S-03 | 登录接口无速率限制，可被暴力破解 | 高 | A07: 身份识别和认证失败 |
| S-04 | JWT Token 存储于 localStorage，易受 XSS 攻击窃取 | 高 | A03: 注入 |
| S-05 | 前端离线模式绕过认证（main.js 第71行：本地验证 admin/admin123 后设置 `offline-token`） | **严重** | A01: 权限控制失效 |
| S-06 | Chart.js 通过 CDN 引入无 SRI（Subresource Integrity）校验 | 中 | A08: 软件和数据完整性失败 |
| S-07 | 无 Content-Security-Policy、X-Frame-Options 等安全响应头 | 中 | A05: 安全配置错误 |
| S-08 | 后端异常处理中 `str(e)` 直接返回给客户端（app.py 第198行），泄露内部信息 | 中 | A09: 安全日志和监控失败 |
| S-09 | admin.html 仅前端 Token 检查，无服务端页面级鉴权 | 高 | A01: 权限控制失效 |
| S-10 | POST `/api/metrics` 未对 key 做白名单校验，可注入任意字段 | 中 | A03: 注入 |

### 1.3 性能层面

| 问题编号 | 问题描述 | 影响 |
|---------|---------|------|
| P-01 | 无 HTTP 缓存策略（Cache-Control / ETag），每次访问重复请求全部数据 | 带宽浪费、响应慢 |
| P-02 | Render 免费版 15 分钟无请求后休眠，冷启动延迟 30s+ | 首次访问体验差 |
| P-03 | 无静态资源 CDN 加速，GitHub Pages 在国内访问慢 | 国内用户加载慢 |
| P-04 | Chart.js 全量引入（约 200KB），但仅使用 line 和 doughnut 两种图表 | 首屏加载时间长 |
| P-05 | 无 Gzip/Brotli 压缩配置 | 传输体积大 |
| P-06 | SQLite 无连接池，每次请求新建连接 | 并发性能瓶颈 |
| P-07 | 无数据库索引优化策略 | 数据量增长后查询变慢 |

### 1.4 体验层面

| 问题编号 | 问题描述 | 影响 |
|---------|---------|------|
| U-01 | API 请求无 Loading 状态提示，数据加载时界面显示 `--` | 用户误以为加载失败 |
| U-02 | 版本切换按钮（对外版/对内版）仅切换高亮样式，未实际过滤内容 | 功能失效 |
| U-03 | 无 Skeleton 骨架屏或占位动画 | 首屏白屏感 |

| U-05 | 错误提示不友好，API 异常时静默降级为默认数据，用户无感知 | 数据可信度降低 |
| U-06 | 移动端流程步骤箭头旋转90度后布局错位 | 移动端体验差 |
| U-07 | admin.html 无登录态过期提醒，Token 过期后操作均静默失败 | 管理体验差 |

### 2.2 技术选型对比

| 层级 | V1.0 技术 | V2.0 技术 | 升级理由 |
|------|----------|----------|---------|
| **前端** | HTML + CSS + JS + Chart.js(CDN全量) | HTML + CSS + JS + Chart.js(本地Tree-shaken) | 减少外部依赖风险，降低首屏体积 |
| **后端** | Python Flask | Python Flask + Flask-Limiter + Flask-Migrate | 增加速率限制、数据库迁移能力 |
| **数据库** | SQLite（临时文件系统） | PostgreSQL（Render/Supabase 托管） | 数据持久化、并发支持、云端管理 |
| **部署** | GitHub Pages + Render Free | Vercel/Netlify + Render Starter | 国内加速、消除冷启动、HTTPS 自动配置 |
| **认证** | JWT + 硬编码密码比对 | JWT + bcrypt 哈希 + HttpOnly Cookie | 消除凭据泄露和 XSS Token 窃取风险 |
| **监控** | 无 | Sentry + 结构化日志 | 线上异常可追溯、主动告警 |
| **缓存** | 无 | HTTP Cache-Control + ETag | 减少重复请求，提升加载速度 |

## 三、安全架构设计

### 3.1 认证与授权体系

```
┌─────────────────────────────────────────────────────────┐
│                    认证流程（改进后）                      │
│                                                         │
│  用户输入凭据 ──► 后端 bcrypt 验证 ──► 签发 JWT          │
│                                                         │
│  JWT 存储方式: HttpOnly + Secure + SameSite Cookie       │
│  （替代 localStorage，防止 XSS 窃取）                     │
│                                                         │
│  Token 策略:                                             │
│  - Access Token: 有效期 2 小时                           │
│  - Refresh Token: 有效期 7 天，HttpOnly Cookie            │
│  - 刷新机制: Access Token 过期后自动用 Refresh Token 续签  │
└─────────────────────────────────────────────────────────┘
```

**用户角色与权限矩阵（RBAC）**

| 角色 | 公开指标(GET) | 内部指标(GET) | 指标修改(POST) | 管理后台 | 审计日志 |
|------|:---:|:---:|:---:|:---:|:---:|
| 外部访客 | Y | N | N | N | N |
| 内部员工 | Y | Y | N | N | N |
| 管理员 | Y | Y | Y | Y | Y |

### 3.2 安全加固清单

| 序号 | 加固项 | 具体措施 | 对应 V1.0 问题 |
|------|-------|---------|---------------|
| 1 | 密钥管理 | SECRET_KEY 通过环境变量注入，禁止代码中硬编码；使用 `secrets.token_hex(32)` 生成 | S-01 |
| 2 | 密码存储 | 使用 `bcrypt` 哈希存储密码，数据库中仅存哈希值；用户表增加 `password_hash` 字段 | S-02 |
| 3 | 登录防暴力破解 | Flask-Limiter 限制登录接口：同一 IP 每分钟最多 5 次，连续失败 10 次锁定 30 分钟 | S-03 |
| 4 | Token 安全存储 | JWT 通过 HttpOnly + Secure + SameSite=Strict Cookie 传输，前端不再 localStorage 存储 | S-04 |
| 5 | 消除前端认证绕过 | 删除 main.js 中离线模式本地密码验证逻辑；离线时仅展示公开数据，不展示内部数据 | S-05 |
| 6 | SRI 校验 | CDN 引入的外部脚本添加 `integrity` 和 `crossorigin` 属性 | S-06 |
| 7 | 安全响应头 | 配置 CSP、X-Frame-Options、X-Content-Type-Options、Strict-Transport-Security | S-07 |
| 8 | 异常信息脱敏 | 生产环境异常返回统一错误码 + 通用提示，详细错误仅写入服务端日志 | S-08 |
| 9 | 接口输入校验 | POST 请求增加 key 白名单校验 + 数值范围校验，拒绝非法字段 | S-10 |
| 10 | CORS 白名单 | 仅允许指定前端域名（如 `https://kingdee-quality.vercel.app`）跨域请求 | A-01 |

### 3.3 安全响应头配置示例

```python
@app.after_request
def set_security_headers(response):
    response.headers['Content-Security-Policy'] = (
        "default-src 'self'; "
        "script-src 'self' https://cdn.jsdelivr.net; "
        "style-src 'self' 'unsafe-inline'; "
        "img-src 'self' data:; "
        "connect-src 'self' https://your-api-domain.onrender.com"
    )
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['X-XSS-Protection'] = '1; mode=block'
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'
    response.headers['Referrer-Policy'] = 'strict-origin-when-cross-origin'
    return response
```

## 四、性能优化方案

### 4.1 前端性能优化

| 优化项 | 当前状态 | 优化方案 | 预期效果 |
|-------|---------|---------|---------|
| **Chart.js 体积** | CDN 全量 ~200KB | 仅引入 line + doughnut 模块（Tree-shaken 构建），或使用轻量替代方案（如 uPlot ~35KB） | 首屏 JS 体积减少 60%+ |
| **首屏加载** | 无优化 | 关键 CSS 内联、非关键资源延迟加载（`defer`/`async`） | FCP 减少 0.5-1s |
| **骨架屏** | 无 | 数据加载期间展示 Skeleton 占位动画 | 消除白屏感知 |
| **静态资源缓存** | 无缓存策略 | 文件名 Hash（`style.a3b2c1.css`）+ 长期缓存（`Cache-Control: max-age=31536000`） | 重复访问零网络请求 |
| **图片优化** | SVG 内联 | SVG 已较优；如需位图场景，使用 WebP 格式 + 响应式图片 | 图片体积减少 30-50% |

### 4.2 后端性能优化

| 优化项 | 当前状态 | 优化方案 | 预期效果 |
|-------|---------|---------|---------|
| **数据库连接** | 每次请求新建 SQLite 连接 | PostgreSQL + 连接池（SQLAlchemy + `pool_size=5`） | 消除连接建立开销 |
| **API 响应缓存** | 无 | 公开指标接口增加 `Cache-Control: public, max-age=300`（5 分钟缓存） | 减少数据库查询 80%+ |
| **ETag 支持** | 无 | 基于数据内容 Hash 生成 ETag，304 响应节省传输 | 带宽减少 90%（缓存命中时） |
| **冷启动优化** | Render Free 15min 休眠 | 方案 A：升级 Render Starter 消除休眠；方案 B：UptimeRobot 保活 + 健康检查端点 | 首次访问延迟从 30s 降至 <2s |
| **响应压缩** | 无 | Flask 配置 Gzip 压缩（`flask-compress`） | JSON 响应体积减少 60-70% |

### 4.3 性能指标目标（Core Web Vitals）

| 指标 | 当前估计 | V2.0 目标 | 达标标准 |
|------|---------|----------|---------|
| **LCP**（最大内容绘制） | ~3.5s | < 2.5s | Good |
| **FID**（首次输入延迟） | ~150ms | < 100ms | Good |
| **CLS**（累积布局偏移） | ~0.15 | < 0.1 | Good |
| **TTFB**（首字节时间） | ~2s（冷启动30s+） | < 800ms | Good |

## 七、数据库设计（V2.0）

### 7.1 表结构

```sql
-- 用户表
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,       -- bcrypt 哈希
    role VARCHAR(20) NOT NULL DEFAULT 'viewer', -- viewer / editor / admin
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 指标表（支持历史记录）
CREATE TABLE metrics (
    id SERIAL PRIMARY KEY,
    category VARCHAR(20) NOT NULL,             -- public / internal
    key VARCHAR(50) NOT NULL,
    value NUMERIC NOT NULL,
    updated_by INTEGER REFERENCES users(id),
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(category, key)
);

-- 指标变更历史
CREATE TABLE metrics_history (
    id SERIAL PRIMARY KEY,
    metric_id INTEGER REFERENCES metrics(id),
    old_value NUMERIC,
    new_value NUMERIC NOT NULL,
    changed_by INTEGER REFERENCES users(id),
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    change_reason TEXT
);

-- 审计日志
CREATE TABLE audit_logs (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    action VARCHAR(50) NOT NULL,               -- login / logout / update_metric / ...
    resource VARCHAR(100),
    ip_address VARCHAR(45),
    user_agent TEXT,
    details JSONB,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```
